Pattern Overviewنظرة عامة على الباترن

Dependency Injection (DI) is a technique where an object receives its dependencies from an external source rather than creating them itself.Dependency Injection (DI) هي تقنية بيتلقى فيها الـ object الـ dependencies بتاعته من مصدر خارجي بدل ما ينشئها بنفسه.

It implements the Dependency Inversion Principle (the "D" in SOLID): high-level modules should depend on abstractions, not concrete implementations.بينفذ Dependency Inversion Principle (الـ "D" في SOLID): الـ modules عالية المستوى لازم تعتمد على abstractions، مش على implementations كونكريت.

Three Types: Constructor injection (most common), Method injection (route/controller methods), Property injection (not recommended).ثلاث أنواع: Constructor injection (الأكثر شيوعاً)، Method injection (routes/controllers)، Property injection (مش مُوصى به).

Real Problem Exampleمثال على مشكلة حقيقية

Without DI, classes create their own dependencies — tightly coupled to specific implementations. Testing is impossible without the real service. You can't swap Stripe for a mock in tests.من غير DI، الكلاسات بتنشئ الـ dependencies بنفسها — مرتبطة بإحكام بـ implementations محددة. الاختبار مستحيل من غير الخدمة الحقيقية. ما تقدرش تبدل Stripe بـ mock في الـ tests.

Laravel Implementationالتطبيق في Laravel

Laravel's Service Container is its DI engine. When you type-hint a class in a constructor, the container uses PHP Reflection to automatically resolve and inject it — called auto-wiring.الـ Service Container في Laravel هو محرك الـ DI بتاعه. لما تعمل type-hint لكلاس في constructor، الـ container بيستخدم PHP Reflection عشان يحله ويحقنه تلقائياً — بيتسمى auto-wiring.

The container resolves: concrete classes (auto-wired), interfaces (when bound), and factory closures.الـ container بيحل: كلاسات concrete (auto-wired)، interfaces (لما تكون مربوطة)، وfactory closures.

Laravel Code Exampleمثال كود Laravel

Without vs With DI / من غير مقابل مع DI
<?php
// WITHOUT DI (tightly coupled, hard to test)
// من غير DI (مرتبط بإحكام، صعب الاختبار)
class OrderService {
    private StripePayment $payment;
    public function __construct() {
        $this->payment = new StripePayment(config('stripe.key')); // hard-coded!
    }
}

// WITH DI (loosely coupled, testable)
// مع DI (مفكوك، قابل للاختبار)
class OrderService {
    public function __construct(
        private PaymentInterface        $payment,
        private UserRepositoryInterface $users,
        private NotifierInterface       $notifier,
    ) {} // No 'new' keywords — container provides everything / مفيش new — الـ container بيوفر كل حاجة
}
Constructor & Method Injection
<?php
// Constructor Injection (most common / الأكثر شيوعًا)
class OrderController extends Controller
{
    public function __construct(private OrderService $orderService) {}

    // Method Injection — container injects Request automatically
    // Method Injection — الـ container بيحقن Request تلقائياً
    public function store(Request $request): JsonResponse
    {
        $order = $this->orderService->processOrder($request->validated());
        return response()->json($order, 201);
    }
}

// Route closure injection / حقن في Route closure
Route::get('/stats', function (StatsService $stats) {
    return $stats->getSummary(); // Container injects StatsService / الـ container بيحقن StatsService
});
Bindings + Unit Test
<?php
// Bind interfaces in ServiceProvider
$this->app->bind(PaymentInterface::class, StripePayment::class);
$this->app->singleton(UserRepositoryInterface::class, EloquentUserRepository::class);
$this->app->bind(NotifierInterface::class, function ($app) {
    return new EmailNotifier($app->make(\Illuminate\Mail\Mailer::class));
});

// Unit test — inject mocks / Unit test — احقن mocks
class OrderServiceTest extends TestCase
{
    public function test_processes_order(): void
    {
        $payment  = Mockery::mock(PaymentInterface::class);
        $users    = Mockery::mock(UserRepositoryInterface::class);
        $notifier = Mockery::mock(NotifierInterface::class);

        $users->shouldReceive('find')->andReturn(new User());
        $payment->shouldReceive('charge')->once()->andReturn((object)['id' => 'ch_123']);
        $notifier->shouldReceive('send')->once();

        // No database, no Stripe, no emails!
        // مفيش database ولا Stripe ولا emails!
        $service = new OrderService($payment, $users, $notifier);
        $order   = $service->processOrder(['user_id' => 1, 'total' => 99.99]);
        $this->assertInstanceOf(Order::class, $order);
    }
}

When to Use Itمتى تستخدمه

✓ Good Use Cases✓ حالات جيدة

  • Any class with external dependencies
  • Services that need to be unit testable
  • When you want to swap implementations
  • Always use instead of hardcoded new
  • أي كلاس عنده dependencies خارجية
  • Services محتاجة unit testing
  • لما عايز تبدل الـ implementations
  • دايماً بدل الـ new المكتوبة بشكل ثابت

✗ Avoid When✗ تجنبه عندما

  • Simple value objects with no external deps
  • Data Transfer Objects (DTOs)
  • Value objects بسيطة من غير external deps
  • Data Transfer Objects (DTOs)

When NOT to Use Itمتى لا تستخدمه

Not every class needs DI. A Money value object, a DTO, or a simple data transformer with no external dependencies doesn't need to be injected — just instantiate directly.مش كل كلاس محتاج DI. Money value object أو DTO أو data transformer بسيط من غير external dependencies مش محتاج حقن — انشئه مباشرة.
