Pattern Overviewنظرة عامة على الباترن

The Observer pattern establishes a subscription mechanism so multiple objects can automatically react when another object changes state.باترن Observer بيقيم آلية اشتراك عشان objects متعددة تقدر تتفاعل تلقائياً لما object تاني يغير حالته.

Key Idea: "User registered" → automatically: send welcome email, add to CRM, send Slack alert, grant credits. Each action is a separate, decoupled observer.الفكرة الأساسية: "اتسجّل user" → تلقائياً: بعت welcome email، أضفته للـ CRM، بعت Slack alert، منحت credits. كل action هو observer منفصل ومفكوك.

Real Problem Exampleمثال على مشكلة حقيقية

Without Observer, UserService::register() grows with every new requirement: send email, notify admin, track analytics, update CRM... It becomes a monolith.من غير Observer، UserService::register() بتكبر مع كل متطلب جديد: إرسال email، إخطار admin، تتبع analytics، تحديث CRM... بتبقى monolith.

With Observer, the service just fires event(new UserRegistered($user)) and each concern handles itself independently.مع Observer، الـ service بتطلق event(new UserRegistered($user)) بس وكل concern بيعالج نفسه باستقلالية.

Laravel Implementationالتطبيق في Laravel

Laravel has two Observer implementations: (1) Events & Listeners — application-level, can be queued, registered in EventServiceProvider. (2) Eloquent Model Observers — react to model lifecycle: created, updated, deleted.Laravel عندها تطبيقين لـ Observer: (1) Events & Listeners — على مستوى التطبيق، ممكن تتحول لـ queue، متسجلة في EventServiceProvider. (2) Eloquent Model Observers — بتتفاعل مع lifecycle الـ model: created, updated, deleted.

Laravel Code Exampleمثال كود Laravel

app/Events/UserRegistered.php
<?php
namespace App\Events;
use App\Models\User;
use Illuminate\Foundation\Events\Dispatchable;
use Illuminate\Queue\SerializesModels;

// The "Subject" — carries data all observers need
// الـ "Subject" — بيحمل البيانات اللي كل الـ observers محتاجينها
class UserRegistered
{
    use Dispatchable, SerializesModels;
    public function __construct(public readonly User $user) {}
}
app/Listeners/
<?php
// Observer 1: Send welcome email (queued) / مراقب 1: إرسال email ترحيبي (في queue)
class SendWelcomeEmail implements ShouldQueue
{
    public function handle(UserRegistered $event): void
    {
        \Mail::to($event->user)->send(new WelcomeMail($event->user));
    }
}
// Observer 2: Track analytics / مراقب 2: تتبع الـ analytics
class TrackUserRegistration implements ShouldQueue
{
    public function handle(UserRegistered $event): void
    {
        \Analytics::track('user_registered', ['user_id' => $event->user->id]);
    }
}
// Observer 3: Grant trial / مراقب 3: منح التجربة المجانية
class GrantTrialCredits
{
    public function handle(UserRegistered $event): void
    {
        $event->user->credits()->create(['amount' => 100, 'reason' => 'welcome_bonus']);
    }
}
app/Providers/EventServiceProvider.php
<?php
// Register: one event → many listeners (one-to-many Observer)
// تسجيل: event واحد → listeners كتير (Observer one-to-many)
protected $listen = [
    \App\Events\UserRegistered::class => [
        \App\Listeners\SendWelcomeEmail::class,
        \App\Listeners\TrackUserRegistration::class,
        \App\Listeners\GrantTrialCredits::class,
    ],
];
// Fire the event — UserService doesn't know about listeners
// إطلاق الـ event — UserService مش عارف بالـ listeners
event(new UserRegistered($user));
app/Observers/UserObserver.php
<?php
namespace App\Observers;
use App\Models\User;

// Model Observer — reacts to Eloquent lifecycle events
// Model Observer — بيرد على أحداث Eloquent lifecycle
class UserObserver
{
    public function created(User $user): void
    {
        \Log::info("New user: {$user->email}");
    }
    public function updating(User $user): void
    {
        if ($user->isDirty('email')) {
            $user->email_verified_at = null; // reset / إعادة تعيين
        }
    }
    public function deleted(User $user): void
    {
        $user->profile()->delete(); // cleanup / تنظيف
    }
}
// Register: User::observe(UserObserver::class);

When to Use Itمتى تستخدمه

✓ Good Use Cases✓ حالات جيدة

  • Multiple reactions to the same event
  • Decoupling side effects from core logic
  • Notifications, analytics, audit logs
  • Reacting to model lifecycle events
  • ردود فعل متعددة على نفس الـ event
  • فصل الـ side effects عن الـ core logic
  • إشعارات وanalytics وsaudit logs
  • الرد على أحداث lifecycle الـ model

✗ Avoid When✗ تجنبه عندما

  • You need guaranteed sequential execution
  • The reaction is critical to the main flow
  • محتاج تنفيذ متسلسل مضمون
  • الرد حرج للـ main flow

When NOT to Use Itمتى لا تستخدمه

Don't over-event everything. If sending a welcome email is required and synchronous, call it directly rather than firing an event and hoping a listener picks it up.ما تعملش events لكل حاجة. لو إرسال welcome email مطلوب وsynchronous، استدعيه مباشرة بدل ما تطلق event وتتمنى إن listener يعالجه.

Debugging Tip: When something runs unexpectedly, check EventServiceProvider and any registered Observers. Hidden side effects are the main downside.نصيحة debugging: لما حاجة تتنفذ بشكل مفاجئ، تحقق من EventServiceProvider وأي Observers مسجلة. الـ side effects المخفية هي العيب الرئيسي.
