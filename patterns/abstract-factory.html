Abstract FactoryAbstract Factory

Produce families of related objects without specifying concrete classes.إنتاج مجموعات من الـ objects المترابطة من غير تحديد الكلاسات الكونكريت.

Pattern Overviewنظرة عامة على الباترن

The Abstract Factory is a "factory of factories." It creates families of related objects — ensuring all produced objects are always compatible with each other.الـ Abstract Factory هو "factory للـ factories." بينشئ مجموعات من الـ objects المترابطة — بيضمن إن كل الـ objects اللي بتنتجها متوافقة مع بعضها.

Key Idea: Pick a "family" (Stripe), and the factory produces all related objects for that family — all guaranteed compatible.الفكرة الأساسية: اختار "مجموعة" (Stripe)، والـ factory بتنتج كل الـ objects المرتبطة بيها — كلها مضمونة متوافقة.

Real Problem Exampleمثال على مشكلة حقيقية

Your app supports Stripe and PayPal. Each has its own Payment processor, Webhook handler, and Refund processor. Abstract Factory guarantees you never mix a StripePayment with a PayPalWebhook.تطبيقك بيدعم Stripe وPayPal. كل واحد عنده Payment processor وWebhook handler وRefund processor. Abstract Factory بتضمن إنك ما تخلطش StripePayment مع PayPalWebhook.

Laravel Implementationالتطبيق في Laravel

Laravel's Manager classes act as Abstract Factories. The database connection factory creates a complete family (Connection, Grammar, QueryBuilder, SchemaBuilder) all matched to the same database type.الـ Manager classes في Laravel بتعمل كـ Abstract Factories. الـ database connection factory بتنشئ مجموعة كاملة (Connection, Grammar, QueryBuilder, SchemaBuilder) كلها متطابقة مع نفس نوع الـ database.

Laravel Code Exampleمثال كود Laravel

app/Contracts/PaymentFactoryInterface.php
<?php
namespace App\Contracts;
// Abstract Factory — defines the family interface
// يعرّف واجهة المجموعة كاملة
interface PaymentFactoryInterface
{
    public function createProcessor(): PaymentProcessorInterface;
    public function createWebhookHandler(): WebhookHandlerInterface;
    public function createRefundProcessor(): RefundProcessorInterface;
}
app/Payments/StripePaymentFactory.php
<?php
// Concrete Factory — Stripe family / مجموعة Stripe
class StripePaymentFactory implements PaymentFactoryInterface
{
    public function createProcessor(): StripeProcessor
    {
        return new StripeProcessor(config('services.stripe.secret'));
    }
    public function createWebhookHandler(): StripeWebhookHandler
    {
        return new StripeWebhookHandler(config('services.stripe.webhook_secret'));
    }
    public function createRefundProcessor(): StripeRefundProcessor
    {
        return new StripeRefundProcessor();
    }
}
// Concrete Factory — PayPal family / مجموعة PayPal
class PayPalPaymentFactory implements PaymentFactoryInterface
{
    public function createProcessor(): PayPalProcessor
    {
        return new PayPalProcessor(config('services.paypal.client_id'));
    }
    public function createWebhookHandler(): PayPalWebhookHandler { return new PayPalWebhookHandler(); }
    public function createRefundProcessor(): PayPalRefundProcessor { return new PayPalRefundProcessor(); }
}
app/Providers/AppServiceProvider.php
<?php
// Bind the right factory based on config
// اربط الـ factory الصح حسب الـ config
$this->app->singleton(PaymentFactoryInterface::class, function () {
    return match (config('services.payment.driver')) {
        'stripe' => new StripePaymentFactory(),
        'paypal' => new PayPalPaymentFactory(),
        default  => throw new \Exception('Unknown driver'),
    };
});

// In controller — no knowledge of Stripe or PayPal
// في الـ controller — مش عارف Stripe ولا PayPal
$processor = $this->paymentFactory->createProcessor();
$chargeId  = $processor->charge($amount, 'USD');

When to Use Itمتى تستخدمه

✓ Good Use Cases✓ حالات جيدة

  • Multiple platform/provider support (Stripe vs PayPal)
  • Theming systems (Light vs Dark families)
  • Cloud provider abstractions (AWS vs GCP)
  • دعم providers متعددة (Stripe مقابل PayPal)
  • أنظمة الـ theming (فاتح مقابل داكن)
  • abstractions لـ cloud providers

✗ Avoid When✗ تجنبه عندما

  • Only one concrete family exists (overkill)
  • Simple Factory Method is sufficient
  • مجموعة concrete واحدة فقط موجودة
  • Factory Method بسيط كافي

When NOT to Use Itمتى لا تستخدمه

If you only have one implementation today and no plans for a second, you're adding abstraction for nothing. Start simple — refactor to Abstract Factory when a second family appears.لو عندك implementation واحد دلوقتي ومفيش خطط لتاني، إنت بتضيف abstraction من غير فايدة. ابدأ بسيط — وعمل refactor لـ Abstract Factory لما تظهر مجموعة تانية.

YAGNI: "You Ain't Gonna Need It." Don't build Abstract Factory speculatively. Add it when you actually have two concrete families.YAGNI: "مش هتحتاجه." ما تبنيش Abstract Factory بشكل استباقي. ضيفه لما يكون عندك مجموعتين فعلاً.
