Factory MethodFactory Method

Pattern Overviewنظرة عامة على الباترن

The Factory Method defines an interface for creating objects, but lets the factory decide which concrete class to instantiate. The caller doesn't need to know the exact class.الـ Factory Method بيعرّف interface لإنشاء الـ objects، لكن بيسيب للـ factory القرار على أي كلاس concrete تنشئه. الـ caller مش محتاج يعرف الكلاس بالظبط.

Key Idea: "I need a payment processor." The factory decides whether to give you Stripe, PayPal, or Razorpay.الفكرة الأساسية: "أنا محتاج payment processor." الـ factory هي اللي تقرر هتديلك Stripe ولا PayPal ولا غيرهم.

Real Problem Exampleمثال على مشكلة حقيقية

Without a factory, your controller has if/else chains for every notification type. A Factory centralizes this decision.من غير factory، الـ controller بيكون فيه if/else طويل لكل نوع notification. الـ Factory بتمركز القرار ده.

Without Factory (messy / فوضوي)
// Bad — controller knows every type / مش كويس
if ($user->prefers_email) {
    $notifier = new EmailNotifier();
} elseif ($user->prefers_sms) {
    $notifier = new SmsNotifier();
}
$notifier->send($message);

Laravel Implementationالتطبيق في Laravel

Laravel's Manager classes (CacheManager, MailManager, QueueManager) use Factory Method: Cache::driver('redis') calls createRedisDriver() internally.Manager classes في Laravel زي CacheManager وMailManager وQueueManager بتستخدم Factory Method: Cache::driver('redis') بتستدعي createRedisDriver() من جوا.

Eloquent Model Factories in tests are also Factory Method — User::factory()->create().Eloquent Model Factories في الـ tests هي كمان Factory Method — User::factory()->create().

Laravel Code Exampleمثال كود Laravel

app/Contracts/NotifierInterface.php
<?php
namespace App\Contracts;
interface NotifierInterface
{
    public function send(string $to, string $message): bool;
}
app/Factories/NotifierFactory.php
<?php
namespace App\Factories;
use App\Contracts\NotifierInterface;
use App\Notifications\{EmailNotifier, SmsNotifier, PushNotifier};

class NotifierFactory
{
    // Factory method — decides which class to create
    // Factory method — بتقرر أي كلاس تنشئ
    public static function make(string $channel): NotifierInterface
    {
        return match ($channel) {
            'email' => new EmailNotifier(),
            'sms'   => new SmsNotifier(),
            'push'  => new PushNotifier(),
            default => throw new \InvalidArgumentException("Unknown: {$channel}"),
        };
    }
}
app/Http/Controllers/NotificationController.php
<?php
class NotificationController extends Controller
{
    public function send(Request $request)
    {
        $channel = $request->user()->notification_channel;
        // Controller doesn't know or care which class it gets
        // الـ Controller مش محتاج يعرف الكلاس اللي هيجيله
        $notifier = NotifierFactory::make($channel);
        $notifier->send($request->user()->contact, 'Your order shipped!');
        return response()->json(['sent' => true]);
    }
}

When to Use Itمتى تستخدمه

✓ Good Use Cases✓ حالات جيدة

  • Multiple implementations of same interface
  • Object creation varies by context or config
  • Multiple payment gateways, storage drivers
  • implementations متعددة لنفس الـ interface
  • إنشاء الـ objects بيتغير حسب الـ config
  • Multiple payment gateways أو storage drivers

✗ Avoid When✗ تجنبه عندما

  • You only ever create one type
  • Simple DI injection is enough
  • بتنشئ نوع واحد فقط طول الوقت
  • الـ DI البسيط كافي

When NOT to Use Itمتى لا تستخدمه

Don't create a factory just for the sake of it. If you always create the same type of object, a factory adds complexity for no gain.ما تعملش factory بس عشان تعملها. لو دايمًا بتنشئ نفس النوع، الـ factory بتضيف تعقيد من غير فايدة.

Overengineering Warning: A factory that only ever creates one class is just a wrapper around new.تحذير من الـ Overengineering: factory بتنشئ كلاس واحد بس هي مجرد wrapper على new.
