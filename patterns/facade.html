Pattern Overviewنظرة عامة على الباترن

The Facade provides a simplified interface to a complex body of code. It doesn't add new functionality — it makes existing functionality easier to use.الـ Facade بيقدم interface مبسّط لكود معقد. مش بيضيف functionality جديدة — بيخلي الـ functionality الموجودة أسهل في الاستخدام.

Laravel's Twist: Laravel Facades are static-looking proxies that resolve to real instances in the Service Container. Cache::get('key') resolves the Cache instance and calls get() on it.الـ Twist في Laravel: الـ Facades في Laravel بتبدو static لكنها فعلاً proxies بتحل لـ instances حقيقية في الـ Service Container. Cache::get('key') بتحل الـ Cache instance وتستدعي get() عليها.

Real Problem Exampleمثال على مشكلة حقيقية

Processing a payment involves: validate card, create payment intent, charge, send receipt, log transaction. Without a Facade, your controller handles all of this. A PaymentFacade wraps it all behind a single charge() call.معالجة payment بتشمل: التحقق من الكارت، إنشاء payment intent، الخصم، إرسال إيصال، تسجيل المعاملة. من غير Facade، الـ controller بيتعامل مع كل ده. الـ PaymentFacade بيلفها كلها خلف call واحد charge().

Laravel Implementationالتطبيق في Laravel

Laravel's built-in facades (Cache, DB, Auth, Storage, Mail) are static proxies to service container instances. Each defines getFacadeAccessor().الـ facades المدمجة في Laravel (Cache, DB, Auth, Storage, Mail) هي static proxies لـ instances في الـ service container. كل واحدة بتعرّف getFacadeAccessor().

They are testable via Cache::fake(), Mail::fake() — which swap the underlying implementation with a test double.ممكن تعملها test عبر Cache::fake() وMail::fake() — اللي بتبدل الـ implementation الحقيقية بـ test double.

Laravel Code Exampleمثال كود Laravel

How Facades Work / كيف تعمل الـ Facades
<?php
// How Laravel Facades work (simplified)
// إزاي الـ Facades في Laravel بتشتغل (مبسّط)

abstract class Facade
{
    abstract protected static function getFacadeAccessor(): string;

    public static function __callStatic(string $method, array $args): mixed
    {
        // 1. Resolve from container / حل من الـ container
        $instance = app(static::getFacadeAccessor());
        // 2. Call method on real instance / استدعي الـ method على الـ instance الحقيقية
        return $instance->$method(...$args);
    }
}
// The Cache Facade / الـ Cache Facade
class Cache extends Facade
{
    protected static function getFacadeAccessor(): string
    {
        return 'cache'; // container binding key
    }
}
app/Services/PaymentService.php
<?php
// Complex subsystem — multiple classes working together
// Subsystem معقد — كلاسات متعددة بتشتغل مع بعض
class PaymentService
{
    public function __construct(
        private CardValidator $validator,
        private StripeClient  $stripe,
        private ReceiptMailer $mailer,
        private PaymentLogger $logger,
    ) {}

    // Facade method: hides all complexity behind one call
    // Facade method: بتخبّي كل التعقيد خلف call واحد
    public function charge(User $user, float $amount): PaymentResult
    {
        $this->validator->validate($user->card);
        $intent = $this->stripe->createPaymentIntent($amount, 'usd');
        $charge = $this->stripe->confirm($intent->id);
        $this->mailer->sendReceipt($user, $amount);
        $this->logger->log($user, $charge);
        return new PaymentResult(success: true, chargeId: $charge->id);
    }
}
app/Facades/Payment.php + Usage
<?php
namespace App\Facades;
use Illuminate\Support\Facades\Facade;

class Payment extends Facade
{
    protected static function getFacadeAccessor(): string
    {
        return 'payment'; // must match container binding
    }
}

// Register in AppServiceProvider:
// $this->app->singleton('payment', PaymentService::class);

// Usage in controller — one clean call
// الاستخدام في الـ controller — call واحد نظيف
$result = Payment::charge(auth()->user(), $request->amount);

// Fakeable in tests / ممكن تعمل Fake في الـ tests
Payment::fake();
Payment::shouldReceive('charge')->once()->andReturn(new PaymentResult(true, 'ch_123'));

When to Use Itمتى تستخدمه

✓ Good Use Cases✓ حالات جيدة

  • Simplifying complex multi-step workflows
  • Creating clean public APIs for subsystems
  • Providing a stable API over a changing subsystem
  • تبسيط workflows معقدة متعددة الخطوات
  • إنشاء public APIs نظيفة للـ subsystems
  • توفير API مستقر فوق subsystem متغير

✗ Avoid When✗ تجنبه عندما

  • The subsystem is already simple
  • You need fine-grained control over each step
  • The Facade becomes a "god object"
  • الـ subsystem بسيط أصلاً
  • محتاج تحكم دقيق في كل خطوة
  • الـ Facade بيبقى "god object"

When NOT to Use Itمتى لا تستخدمه

Watch out for "Facade bloat." If your facade has more than 10-15 methods, consider splitting into smaller facades or services.احترس من "Facade bloat." لو الـ facade عندها أكتر من 10-15 method، فكر في تقسيمها لـ facades أو services أصغر.

Facade vs Helper: cache('key') helper is simpler but not mockable. Prefer Facades in application logic where testability matters.Facade مقابل Helper: helper زي cache('key') أبسط لكن مش قابلة للـ mock. فضّل الـ Facades في الـ application logic لما الـ testability مهمة.
