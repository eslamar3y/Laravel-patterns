Repository PatternRepository Pattern

Abstract your data layer — keep business logic clean and database logic separate.فصل الـ data layer — احتفظ بالـ business logic نظيف والـ database logic منفصل.

Pattern Overviewنظرة عامة على الباترن

The Repository Pattern acts as a collection-like abstraction over your data source. Business logic talks to a repository interface; the repository handles all database queries.باترن Repository بيعمل abstraction شبه collection على الـ data source. الـ business logic بتتكلم مع repository interface؛ والـ repository بتتعامل مع كل استعلامات الـ database.

Key Benefit: Swap your data source (Eloquent → API → Cache → MongoDB) without touching a single line of business logic.الفايدة الرئيسية: بدّل الـ data source (Eloquent → API → Cache → MongoDB) من غير ما تلمس سطر واحد من الـ business logic.

Real Problem Exampleمثال على مشكلة حقيقية

Without Repository, User::where('active', 1)->get() is scattered everywhere. Unit testing requires a database. Switching from MySQL to MongoDB means hunting through dozens of files.من غير Repository، User::where('active', 1)->get() مبعثرة في كل مكان. الـ unit testing محتاج database. التحول من MySQL لـ MongoDB يعني تصيد في عشرات الملفات.

With Repository, services call $this->userRepo->getActiveUsers(). Tests inject a mock. Switching databases means writing one new class.مع Repository، الـ services بتستدعي $this->userRepo->getActiveUsers(). الـ tests بتحقن mock. تبديل الـ database معناه كتابة كلاس جديد واحد بس.

Laravel Implementationالتطبيق في Laravel

Laravel doesn't include Repository classes, but its architecture makes it easy. Define an interface, create an Eloquent implementation, bind them in a Service Provider.Laravel مش عندها Repository classes جاهزة، لكن معمارتها بتسهّل تنفيذها. عرّف interface، انشئ Eloquent implementation، واربطهم في Service Provider.

The container acts as the glue — anywhere you type-hint the interface, the container injects the Eloquent implementation.الـ container بيعمل كالـ glue — في أي مكان تعمل فيه type-hint للـ interface، الـ container بيحقن الـ Eloquent implementation.

Laravel Code Exampleمثال كود Laravel

app/Contracts/UserRepositoryInterface.php
<?php
namespace App\Contracts;
interface UserRepositoryInterface
{
    public function find(int $id): ?User;
    public function findByEmail(string $email): ?User;
    public function getActive(): Collection;
    public function paginate(int $perPage = 15): LengthAwarePaginator;
    public function create(array $data): User;
    public function update(int $id, array $data): bool;
    public function delete(int $id): bool;
}
app/Repositories/EloquentUserRepository.php
<?php
namespace App\Repositories;
use App\Contracts\UserRepositoryInterface;
use App\Models\User;

class EloquentUserRepository implements UserRepositoryInterface
{
    public function find(int $id): ?User
    {
        return User::find($id);
    }
    public function findByEmail(string $email): ?User
    {
        return User::where('email', $email)->first();
    }
    public function getActive(): Collection
    {
        return User::where('active', true)->orderBy('name')->get();
    }
    public function create(array $data): User
    {
        return User::create($data);
    }
    public function update(int $id, array $data): bool
    {
        return User::where('id', $id)->update($data);
    }
    public function delete(int $id): bool
    {
        return User::destroy($id) > 0;
    }
}
Binding & Usage / الربط والاستخدام
<?php
// Bind in ServiceProvider
$this->app->bind(UserRepositoryInterface::class, EloquentUserRepository::class);
// To switch source: change this one line / لتبديل المصدر: غيّر السطر ده بس

// Service depends on INTERFACE only / الـ Service بيعتمد على الـ Interface بس
class UserService
{
    public function __construct(private UserRepositoryInterface $users) {}

    public function registerUser(array $data): User
    {
        $user = $this->users->create([...$data, 'password' => bcrypt($data['password'])]);
        event(new UserRegistered($user));
        return $user;
    }
}

// Unit test — mock, no DB needed / Unit test — mock، مفيش DB
$mock = Mockery::mock(UserRepositoryInterface::class);
$mock->shouldReceive('create')->once()->andReturn(new User());
$service = new UserService($mock);

When to Use Itمتى تستخدمه

✓ Good Use Cases✓ حالات جيدة

  • Large apps where unit testing matters
  • Apps that may change data sources
  • Domain-heavy apps with complex business logic
  • تطبيقات كبيرة حيث الـ unit testing مهم
  • تطبيقات ممكن تغير مصادر البيانات
  • تطبيقات بـ domain logic معقدة

✗ Avoid When✗ تجنبه عندما

  • Simple CRUD apps — adds unnecessary layers
  • Small projects where Eloquent in controllers is fine
  • تطبيقات CRUD بسيطة — بيضيف طبقات مش لازمة
  • مشاريع صغيرة حيث Eloquent في الـ controllers كافي

When NOT to Use Itمتى لا تستخدمه

Many developers argue that Eloquent Models are already repositories — they abstract database access. Adding another Repository layer can just be wrapping one abstraction with another.كتير من المطورين بيقولوا إن Eloquent Models هي repositories أصلاً — بتعمل abstraction للـ database access. إضافة طبقة Repository فوقيها ممكن تكون مجرد تغليف abstraction فوق abstraction.

Laravel Reality: For most projects, Eloquent in Service classes is perfectly clean. Reach for Repository when you have a genuine need — like testing without DB, or multiple data sources.الواقع في Laravel: لمعظم المشاريع، Eloquent في الـ Service classes نظيف كفاية. استخدم Repository لما عندك حاجة حقيقية — زي testing من غير DB أو مصادر بيانات متعددة.
